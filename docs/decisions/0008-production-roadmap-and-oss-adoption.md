# ADR-0008：生产化路线排序与开源引擎采用

- 状态：Accepted
- 日期：2026-07-16
- 决策者：项目组（AI 代理执行）

## 背景（Context）

M0–M6 + EVAL 已交付**离线优先的研究/验证骨架**（125 测试全绿），但离生产级 trading agent 仍有明确差距（见 [PRODUCTION-READINESS.md](../PRODUCTION-READINESS.md)）：真实接入、执行真实性、可观测性/运维、合规。需要决定 **(a) 补齐差距的顺序**、**(b) 是否采用成熟开源引擎**。约束：本机算力有限、默认模拟盘、研究纪律要求"无真实 alpha 证据不重工程"。

## 备选方案（Options）

- 补齐顺序：
  - A1 先做工程/运维（消息总线、K8s、可观测性），再接真实数据/LLM。
  - A2 先接真实数据/LLM 拿 alpha 证据，再逐步工程化。
  - A3 直接上真实资金小额实盘边跑边补。
- 执行引擎：
  - B1 继续自建/扩展 M5 参考引擎与模拟 broker。
  - B2 采用 NautilusTrader（回测-实盘同源、执行真实性、事件溯源）。
  - B3 采用 Lean/QuantConnect（多资产、公司行为强，C# 核较重）。
- 研究加速：
  - C1 只用自建参考回测引擎。
  - C2 引入 vectorbt 做大规模参数扫描/walk-forward。

## 决策（Decision）

- 顺序选 **A2**，落为 **M7 → M8 →（并行 M9）→ M10**：
  1. **M7 真实接入 MVP**（数据 + LLM + AI gateway）——最高杠杆。**没有真实 alpha 证据前，重工程都是过早优化**；若证否 Edge 也是有价值的负结论。
  2. **M8 生产执行引擎**——拿到执行真实性 + 回测-实盘同源。
  3. **M9 可观测性/运维**——可与 M8 并行；碰真钱的硬前置。
  4. **M10 合规 + 上线闸门 + 小额实盘**——前置全绿后逐步放量。
  - 明确**否决 A3**（不确证前上真钱，违反安全红线与研究纪律）。
- 执行引擎选 **B2 NautilusTrader**：以其为运行时，我方 `core` 契约 + `RulesDecisionPolicy` 为资产，做薄适配（决策代码零改）。兑现 [ADR-0002](0002-leverage-oss-vs-build.md)"复用而非手搓"，且直接落地 [ADR-0003](0003-backtest-live-parity.md) 回测-实盘一致。Lean 作多资产备选。
- 研究加速采纳 **C2**：EVAL/M6 可选引入 vectorbt 做参数扫描，但**评测口径（DSR/PBO/holdout）仍以我方 `eval/` 为单一裁判**，避免"漂亮回测"绕过防过拟合门。

## 后果（Consequences）

- 正面影响：资源投在最高不确定性处（真实 alpha）；执行真实性与运维复用成熟方案，少造轮子；接缝（`core` Protocol）让替换是"换实现"而非"重写调用方"。
- 负面影响 / 取舍：Nautilus 学习曲线陡、GPLv3 许可需评估；真实接入引入成本/密钥/合规面；A2 意味着短期内不追求高吞吐架构（低频可接受，非 HFT）。
- 后续需要注意 / 复核：M7 若证否 Edge，应调整假设而非硬推 M8+；Nautilus 集成成本过高时的折中（M5 引擎 + 真实 paper 券商）；vectorbt 结果必须回灌我方 eval 裁定；真实资金仍需 M10 记分卡全绿 + 人类签署。
